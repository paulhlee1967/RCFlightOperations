Membership applications

When applicants use your club’s online form at /apply.php, submissions land in Applications for staff review before anyone is added or updated in the member roster.

Stripe payments on apply.php Paid applications show Stripe totals for review. After approval, use Process Signup / Renewal to record fulfillment the same way you would for any other member (complimentary and coupon waivers are flagged automatically).

Overview

Each website submission creates a pending application row. Staff compare the form data to existing members (AMA number, name, email, birthday), review uploaded file links (badge photo and any FAA registration upload on the website), and decide whether to approve or reject. Approved applications create a new member or update a matched member, copy the badge photo when one was submitted, then hand off to renewal recording.

The navbar badge and dashboard “Needs attention” card show how many applications are still pending — counted across all renewal years, not just the year currently shown in the list.

Who can use it

  • Administrator and Membership Manager — full review, approve, and reject.
  • Club Staff — can review and approve (same as renewal processing access).
  • Report Viewer — no access to Applications.

AMA gate & club-record prefill

Before the full application form, applicants enter their AMA number and last name. The app verifies membership against AMA’s public lookup, then applies club rules:

  • Minimum AMA expiration — membership must be valid through your club’s required date (life members exempt).
  • Suspended members — if the AMA # matches a suspended club record, online apply is blocked with a contact-the-club message.
  • Renewal eligibility — during renewal season, a matching current member may choose Renewal; otherwise the form steers them to New member.
  • Prefill from club roster — when the AMA # matches an existing member, editable fields (address, phone, email, birthday, emergency contact, FAA, membership type) are filled from the club record. First name, last name, AMA #, and AMA expiration stay read-only from the AMA verification.
  • Existing badge photo — if the club record has a badge photo, it is shown on the apply form. Applicants may keep it (leave the upload blank) or replace it. New members without a photo on file must upload one.
  • FAA registration upload — must be valid through the same minimum date as AMA. If a registration file is already on file and the expiration meets that minimum, the upload is optional (leave blank to keep it). Otherwise a new file is required.

Applicants can still edit any prefilled value before submit. Staff matching on approve still uses AMA + name/email/birthday tiers for linking to the correct member.

Review workflow

  1. Open Applications from the top nav (or follow the dashboard link).
  2. Select an applicant from the list to open the detail panel.
  3. Check the suggested member match and any Changes vs matched member table.
  4. Review payment info, email preferences, and uploaded file links (badge photo is copied to the member record on approve).
  5. Use Request information if you need more details from the applicant (application stays pending).
  6. Adjust Renewal type and Renewal year if needed.
  7. Click Approve & process or Reject.

Approval redirects to Process Signup / Renewal for the member so you can record payment and print a badge. If a badge photo was on the application, it should already be on the member record — go straight to fulfillment when ready. See Renewals & Dues and Badge printing.

Filters & pagination

The list is designed for ongoing season work, not an endless archive of every submission.

ControlBehavior
Status tabs Pending (default), Approved, Rejected, or All
Renewal year Defaults to the club’s current renewal year (same logic as Process Signup / Renewal). Choose All years to browse history.
Search Name, email, or submission reference
Reset Clears search and returns to the current renewal year
Pagination 50 applications per page; Prev/Next when there are more

If you are filtered to one renewal year but pending applications exist in other years, a yellow banner appears with a link to Show all years so older pending items are not overlooked.

Payment display

The payment section summarizes what the applicant paid on apply.php:

  • Membership dues — calculated from membership type and renewal season
  • Initiation and Processing fee (Stripe)
  • Subtotal — when all line items are present
  • Coupon / complimentary — from notes or waived payment status
  • Total paid — Stripe charge when payment was required

Email preferences

The public form at apply.php includes an Optional email notifications section with two independent checkboxes (both unchecked by default):

  • Club events & announcements — occasional fly-ins, meetings, and field updates via Sender’s campaign channel. Leaving this unchecked does not remove someone who already receives club mail from another sign-up; they can use the unsubscribe link in those messages.
  • AMA & FAA expiration reminders — scheduled expiry notices via Sender’s transactional channel (separate from club event mail). Each reminder includes its own opt-out link on unsubscribe.php. If unchecked, the app will not send reminders and will opt the member out of transactional mail in Sender when approved.

Staff see the applicant’s choices in the detail panel under Email preferences (or “No optional emails selected”). On Approve & process, preferences are copied to the member record and synced to Sender.net when configured under Administration → Installation (see Sender.net opt-out).

Club-event opt-in is also pushed to Sender when the application is first submitted (if that box was checked). Expiry-reminder preference is applied on approve so staff review completes before transactional status changes.

Applicant emails

Branded status emails are sent automatically at key points in the application lifecycle. Club name and contact details come from Configuration and Administration → Installation — nothing is hardcoded to a specific club.

  • Application received — sent when the submission is finalized (payment confirmed or waived). Duplicate sends are prevented atomically; if SMTP fails, the detail panel shows a failed status with a Retry received email button.
  • Application approved — sent after staff approve the application. Approval succeeds even when email fails; use Retry approval email if needed.
  • Request information — staff enter a required message; the application stays pending (no separate status). The latest message and full history appear in the detail panel. Accidental double-clicks on the same message are deduplicated for about two minutes; you can send new requests later.

Approve, reject, and request-information actions are recorded in the audit log (Administrator).

Approve & reject

Approve

  • Matched member — updates only non-empty fields from the form (does not wipe existing data).
  • No match — creates a new member record.
  • Ambiguous match — pick the correct member from the dropdown before approving.

Reject

Rejecting moves the application off the Pending tab and keeps it under Rejected with an optional reason. Rejected rows remain in the database for reference. No automatic email is sent on reject.

Request information

Open Request information, enter a required message, and click Send request. The applicant receives a branded email asking them to reply to the club support address. The application remains pending until you approve or reject it.

Uploaded files

Applicants upload a badge photo and FAA registration on apply.php. Approving the application copies those files onto the member record for badge printing and the Compliance tab.

  • Badge photo: JPEG or PNG (GIF also accepted).
  • FAA registration: PDF, JPEG, or PNG.
  • Failed copy — approval still succeeds; upload manually on the member record if needed.
Test submissions There is no delete button in the app yet. To fully remove a test application (for example so the same submission reference cannot be reused), an admin must delete the row from the member_applications table in the database. Rejecting alone does not remove it.

Public form setup

Configure Stripe keys and the Application signing secret under Administration → Installation. Link your club website to /apply.php on this server. Stripe webhooks should target api_stripe_webhook.php.

Complimentary invites

Staff can create complimentary membership invites (email and/or AMA number, free membership or life member) from the complimentary invites screen. When an applicant matches an open invite on apply.php, payment is waived and the invite is marked redeemed on successful submit. Life/free flags already on a club member record can also waive payment without a separate invite.

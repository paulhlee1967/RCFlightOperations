Administration

Manage who can log in and what they can do, customize your club's branding, and configure dues rates.

Most of this section requires the Administrator role. The Badge Designer is also accessible to Membership Managers. Other roles won't see the Administration menu.

Users & Roles

Go to Administration → Users. This lists every account that can log into RC Flight Operations for your club. Each account has an email (used to log in), a display name, a role, and an active/inactive flag.

Users are separate from members — a board member who logs in to run reports has a user account, but may or may not also be in the member database.

Adding a User

  1. Go to Administration → Users → Add user.
  2. Enter their email address — this is what they use to log in.
  3. Enter their display name (shown in the top-right nav badge).
  4. Choose a role (see below).
  5. Set a temporary password — they should change it after first login.
  6. Make sure Active is checked.
  7. Save.

To deactivate a user (e.g., board member who has left), edit their account and uncheck Active. Their login is disabled but their account is kept for audit history.

Club Configuration

Go to Administration → Configuration. Settings are organized into three tabs: General, Design, and Membership & Dues.

Separately, Administration → Installation covers app-wide host settings organized in tabs (General, Applications, Email, Board packet, Tools & status): SMTP and Sender.net reminder opt-out (see below), Renewal year default — pre-book starts (first month when “next calendar year” becomes the default renewal year in Process Signup / Renewal; default October), Reports — complete data starting year, monthly board packet recipients, and Membership application (Stripe) keys and signing secret for the public apply.php form — see Applications — Public form setup. The pre-book month also drives the default Renewal year filter on Website Applications.

Membership & Dues

The Membership & Dues tab lets you set rates per membership type. Each type (Adult, Youth, Senior, Spouse) has independent settings:

SettingDescription
Annual duesFull-year renewal amount in dollars
Initiation feeOne-time fee added to a new member's first payment (set to 0 if not applicable)
Late feeAdded when a renewal is recorded as "Late"
Prorate monthsMembers joining during the last N months of the year pay a prorated amount. Set to 0 to disable proration.

See Renewals — Configuring Dues for more detail on how proration is calculated.

Email Configuration

Prefer Administration → Installation → Email in the app to test email and review SMTP-related options when your build exposes them. Under the hood, outgoing mail is still driven by config.php (SMTP or PHP mail()) on most installs. The app uses it for Email report (one formatted message to an address you choose), Email members (individual messages per member from selected reports), and other features that send mail.

By default the app uses the server's built-in mail, which works on many hosts but sometimes lands in spam.

For reliable delivery, your club's webmaster needs to configure SMTP (outgoing mail) in the server configuration. Good free options for club volumes:

  • Brevo (formerly Sendinblue) — free up to 300 emails/day, SMTP included
  • Gmail SMTP — use a club Gmail account with an App Password
  • Your host's SMTP — most cPanel hosts provide an outgoing mail server

If you don't have access to the server config, ask whoever installed RC Flight Operations for your club to set up SMTP. Technical documentation in the project repository includes an example configuration for developers.

Sender.net Opt-Out (Scheduled Reminders)

If your club uses Sender.net for newsletters and member list management, configure Administration → Installation → Email → Sender.net (reminder opt-out):

  • API access token — from Sender → Settings → API access tokens. Used to ensure subscribers exist (lowercase email), check transactional (reminder) status, assign the members group, and send reminders via Sender’s API.
  • Members group ID — Sender group for auto-added subscribers (Subscribers → your members list → group settings). Required for reminders.
  • Canonical host — set canonical_host (or public_base_url) in config.php on the server so cron-built emails include working logo and unsubscribe links.

Two separate channels: Club events use Sender’s campaign channel (status.email). AMA/FAA expiry reminders use the transactional channel (status.temail). Unsubscribing from newsletters does not stop reminders, and vice versa.

How preferences are set:

  • Public application (apply.php) — optional checkboxes for club events and expiry reminders. Club-event opt-in syncs to Sender on submit; both preferences sync on staff approve. See Applications — Email preferences.
  • Member contact tab — when Sender is configured, shows a read-only campaign/transactional status panel with a link to open the contact in Sender.net. See Members — Contact tab.
  • Reminder-only unsubscribe — signed link in each expiry email (unsubscribe.php) sets transactional status to UNSUBSCRIBED without affecting campaign mail.
  • Existing members — members.email_opt_in_expiry_reminders defaults to opted in for databases migrated before this feature; send_reminders.php skips members who opted out on file. Emails are stored lowercase to avoid duplicate subscribers.

Test without sending: php scripts/send_reminders.php --dry-run (from SSH on your host). Use --test-email=you@example.com with optional --test-limit=3 to receive sample reminders. Use --dump-sender-payload to inspect the Sender API body (token redacted).
